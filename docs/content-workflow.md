# コンテンツ更新方針

## 自動更新の基本線

毎日6:00（日本時間）にナレッジ、Webニュース、ショート動画台本を同時に下書き生成します。公開前に必ず人が確認し、レビューPRのマージを承認操作として扱います。

| コンテンツ | 頻度 | 自動生成 | 公開条件 |
| --- | --- | --- | --- |
| ナレッジ | 毎日1件 | 本文、用語集、クイズ、SEO、参考情報 | PRで事実・重複・表現を確認してマージ |
| Webニュース | 毎日1件 | 一次情報を起点に変更点、影響、対応を整理 | PRで発表日、URL、影響範囲を確認してマージ |
| ショート動画 | 毎日1本 | 当日の記事から30〜60秒の台本を生成 | PRでフック、説明、CTAを確認してマージ |
| 基礎知識 | 必要時に更新 | 既存の学習体系を保ちながら追記 | 全体の重複と学習順を確認してマージ |

## 状態遷移

1. `draft`: AIが生成し、日付別PRで確認中
2. `approved`: PRを人がマージし、承認者と承認日時を記録
3. `published`: 公開用インデックスと記事JSONをmainへ追加
4. Vercel: mainの公開コミットを自動デプロイ

PRを閉じただけの場合は承認も公開も行いません。生成処理が失敗した場合も、途中データをmainへ直接追加しません。

## 品質チェック

- JSON形式、必須本文、カテゴリ、slug、読了時間
- ナレッジの重要ポイント、用語集、クイズ
- ニュースの要点3件、変更内容、影響、対応
- HTTPSの参考URL、ニュースの一次情報
- 既存記事とのslug・タイトル衝突
- 下書きの未承認状態と承認後の監査情報
- Next.jsの本番ビルド

## 運用上の原則

- AIは公開権限を持たず、人のPRマージを必須にする。
- ニュースは本数より一次情報、鮮度、対象読者への影響を優先する。
- 確認できない内容は断定せず、限定的なニュースを無理に選ばない。
- ショート動画は通常ナレッジを使い、緊急度4以上のニュースのみ優先する。
- APIキーはGitHub Secretsだけで管理し、ソースコードへ保存しない。
